# AgentMesh Runtime — 技术尽调与综合结论

> **Errata（2026-06-22）**：§1 表格里关于「SQLite FTS 名不副实」一条是错的，详见 frontmatter 中 errata 字段。其余结论不受影响（向量层 O(n)、决策智能为空、ranking 是可复制启发式 等均经 Codex 二次代码核对确认）。

## 0. 为什么写这份(以及为什么前两份会分歧)

此前有两份方案:产品化方案(落地路径)和 Codex 的《商业化架构分析》(平台愿景)。它们在**护城河放哪、产品定位、执行路径**上正面对立。

诊断对了一件事:**分歧的根源不是观点之争,而是双方掌握的代码事实不一致**——产品化方案此前只读了部分文件,Codex 基于的是更早的代码快照。所以这次先做**三路并行代码级尽调**(逐行读 4682 行 src + scripts,全部要求引用 `文件:行号`),把"代码现在到底实现了什么"钉死,再在事实地基上裁决分歧。下面每个结论都有代码证据,刻意避免架构愿景词。

## 1. 代码尽调事实(地基)

逐行读完 14 个文件后的实现程度真相:

| 能力 | 实现程度 | 关键事实(证据) |
|---|---|---|
| **多后端 failover** | ✅ 真实现 | `unified_memory_recall.py:383-475` recall() 真串联 Neo4j→SQLite→grep→向量,每层独立 try/except,任一挂掉其余继续 |
| **recall ranking** | ⚠️ 真实现但**无算法壁垒** | `score_text_match:126-189` 是调过参的多因子启发式(短语+2.0/token+1.5/recency+0.18…)。有领域 know-how,但**任何工程师看懂即可复制** |
| **向量层** | ⚠️ 真实现但有硬伤 | Gemini `gemini-embedding-001` 3072 维,`curl` 调用(非 SDK);**O(n) 纯 Python 暴力余弦搜索**(`vector_store.py:262-283`),量大即瓶颈;含 DoH/`198.18.` 本机 hack(`39-62`) |
| **"SQLite FTS"** | ❌ 名不副实 | 实为 `LIKE %token%`(`recall_sqlite:294-301`),不是 FTS5。宣传需改口 |
| **ingest + sync ledger** | ✅ 真闭环 | `append_ledger_event` 在 `episode_ingest.py:208/230/455/512` 真调用;topic/entity 是规则字典匹配,summary 是截断非生成 |
| **一致性恢复栈** | ✅✅ 最扎实(宝石) | deferred sync(`sync_backfill.py:50-110` 真重放)+ checkpoint + rehydrate 全闭环,**无 LLM 依赖、环境耦合最少**,最可直接商业化 |
| **OODA 状态机** | ⚠️ 骨架真,**决策智能是空的** | `autonomous_loop.py:240-265` next-step 来自预写 `goal_frame.steps`(规则推进);Verify 是**字符串包含匹配**(`288-296`);**全栈不自调 LLM**(grep 验证为空) |
| **反模式检测** | ✅ 真实现(规则) | blind_retry/phantom_progress/pivot_exhaustion 是计数器+阈值(`ooda-driver.py:180-280`),能跑,但属明确规则可复制 |
| **first-principles "runtime"** | ❌ 名实不副 | 只有一个 prompt 模板 + recall 轻微加权,**代码层几乎不存在**。这个卖点是方法论不是软件 |
| **heartbeat 6 项自检** | ✅ 真探测 | `heartbeat_systemic.py:80-260` 真连 bolt、查 crontab、测新鲜度 |

**两个贯穿全栈的事实**:
- **A. 这套代码自身几乎不调 LLM。** 唯一的外部 AI 调用是 Gemini embedding。"智能"(决策、验证)预期由**外部 agent**(当前是 OpenClaw)消费 prompt 来提供。它本质是**给一个外部 LLM agent 用的运行时脚手架**,不是自包含 AI 产品。
- **B. 强 OpenClaw + cron 耦合。** `~/.openclaw` 路径、`main/growth/invest` agent 名写死(记忆栈 12 处 + 执行栈多处);`module-5` 整篇死绑 OpenClaw 启动 hook;三个 scripts 假设被 cron 驱动。

**打包/工具链事实**:
- 当前 = pip + setuptools + argparse;`py-modules` 漏了 `journal_to_episode.py`、`kb_to_graph.py`(安装后不可 import)。
- **生态范本(`AgentMesh-Lecturecast`、`lecturecast-server`)已全面 uv + hatchling + typer,且 lecturecast-server 有 `uv.lock`。** 要对齐生态必须迁这三样。
- **core 接入范式现成可照抄**:`lecturecast-server/app/core_client.py` 的 `resolve_api_key` / `deduct` / `refund` + `X-Service-Token` 头,对应 core 真实端点 `POST /v1/internal/credentials/resolve`、`/v1/internal/credits/deduct`(余额不足返 402)、`/refund`。Codex 凭空设计的 server 组件里,鉴权计费这块**不用重新设计**。

## 2. 用事实裁决三个分歧

### 分歧 1 — 护城河放哪 / 开源给多少

- Codex(§4-5):ranking 策略、OODA policy、verification 逻辑是核心护城河,应**留 server、CLI 不暴露**。
- 产品化方案:用户拍板**完整开源**,核心算法也开源,护城河 = 数据 + 托管。

**事实裁决**:Codex 的前提不成立。代码现实是——
1. ranking 是**可复制的启发式调参**(无壁垒);
2. **"OODA Policy Engine" 当前根本不存在**——决策是规则 + 预写 plan,verify 是字符串匹配。

所以"把护城河算法留 server"面对的尴尬是:**目前没有值得藏的算法**。藏一个任何工程师能复制的启发式 ranking,只会牺牲开源引流、换不到真护城河。

→ **维持「完整开源」决策是对的**,且理由比原方案更硬:不是"我们大方",而是"现状没有可藏的算法壁垒,藏了反而吃亏"。真护城河只能是 Codex §4.6 说对的那个——**运行数据飞轮**(用户跑得越多,反模式/recovery/recall 质量样本越多),外加一致性栈的工程完整度和托管便利。开源算法快照 ⊕ server 持续迭代 + 数据飞轮,两者兼得。

### 分歧 2 — 产品定位:平台 vs 工具

- Codex:建"通用 Agent 运行时增强平台"(8 组件 + 5 Adapter + server 决策大脑)。
- 产品化方案:把现有 4 能力分仓 + 双模式 CLI + 托管重依赖。

**事实裁决**:Codex 的愿景方向有洞察(尤其 Agent Adapter 解绑 OpenClaw、observability 当一等公民),但它**严重低估了成本**。事实 A 揭示:当前"智能"是空的,要做成 Codex 的"增强大脑"(server 端 LLM 决策 + 语义 verification),**几乎要从零新建**,不是迁移现有代码。Codex 自己也承认"新建商业化实现"——但没说清现状能复用的只有记忆栈 + 一致性栈(~2000 行),执行层的智能要重写。

→ 正确定位是**两段式**:第一段(现在能卖的)= 把**真实资产**(记忆栈 + 一致性恢复栈)产品化,定位"给你的 agent 加持久记忆 + 执行闭环记录 + 断点恢复";第二段(要新建的)= Codex 的"智能增强大脑"(server 补 LLM 决策/verify),作为路线图而非 MVP。

### 分歧 3 — 执行路径:推倒重建 vs 渐进改造

- Codex(§9):不重构 ARS,**新建**商业化实现。
- 产品化方案:渐进改造现有仓 + 复用生态范本。

**事实裁决**:**混合,且偏渐进**。记忆栈 + 一致性栈是真实可用资产(测过、有 bug 修复记录),推倒是浪费;OODA 决策层是空的、确实该新建(Codex 这半句对)。所以:**渐进复用真资产 + 新建智能层**,不是非此即彼。而且鉴权计费有现成范本可照抄,Codex"从零设计 8 组件"的成本被高估。

## 3. 综合结论

把双方对的部分按事实重新拼装:

| 维度 | 采纳 | 来源 |
|---|---|---|
| 开源策略 | 完整开源(理由升级为"无可藏算法") | 用户决策 + 尽调事实 |
| 护城河 | 运行数据飞轮 + 一致性栈工程度 + 托管 | Codex §4.6 |
| 定位 | 两段式:先记忆+恢复,后智能大脑 | 双方综合 |
| 解绑 OpenClaw | Agent Adapter(OpenClaw/Claude Code/Codex/Hermes/Generic) | Codex §5/§8.3 |
| 产品边界 | 明确"不做":不自动重启/不 daemon/不托管 | Codex §2 |
| 执行 | 渐进复用真资产 + 新建智能层 | 双方综合 |
| 鉴权计费 | 照抄 lecturecast-server core_client 范式 | 尽调事实 |
| 工具链 | **uv + hatchling + typer**(对齐生态) | 尽调事实 |

### 落地路线(基于事实,不画饼)

**Phase 0 — 脱敏 + 工具链对齐(阻塞项)**
- 抽离 12+ 处硬编码(私人实体 `episode_ingest.py:41-63`、降权特例 `unified_memory_recall.py:152-155`、`main/growth/invest`、`~/.openclaw` 路径、DoH hack)→ 改为 env/workspace 配置。
- 工具链迁移:`pip/setuptools/argparse` → **`uv + hatchling + typer`**;补全 `py-modules` 缺的两个文件(或 hatchling 自动发现包)。
- 验证:`uv sync` + `uv run xng demo` 跑通;`grep` 私人标识为空。

**Phase 1 — 抽 `agentmesh_runtime` 包 + 修工程硬伤**
- 记忆栈 + 一致性栈整团进包,引入可注入 `RecallConfig`(多租户)。
- 修事实暴露的硬伤:向量 O(n) 搜索(server 端换 pgvector/faiss)、"FTS" 正名或真上 FTS5。
- 验证:`uv run python -c "from agentmesh_runtime.recall import recall; recall('test')"`;改造前后同 query 输出 diff 一致。

**Phase 2 — 开源 client(先上,引流)**
- 对齐 `AgentMesh-Lecturecast` 骨架,但 client **主打 BYOK 完整本地**(生态首个本地优先 client,有意区别)。
- 卖点诚实:记忆 + 恢复 + loop 记录;**不宣传不存在的 "first-principles runtime"**。
- 验证:`uv run` 全程离线跑通 doctor/recall/loop/rehydrate。

**Phase 3 — 私有 server(从托管重依赖起步)**
- 照抄 `lecturecast-server/app/core_client.py` 接 core(resolve/deduct/refund)。
- 先卖"托管 Neo4j + embedding + 向量索引"(省运维),埋计量点。
- **智能层(LLM 决策 + 语义 verify)在此后新增**——这是 Codex 愿景的真正落点,标记为"新建"。

**Phase 4 — 数据飞轮 + Adapter 扩展**
- 沉淀运行样本(反模式/recovery)→ 反哺策略(真护城河启动)。
- 加 Claude Code / Codex / Hermes Adapter,解绑 OpenClaw。

## 4. 真正该由你拍板的一个问题(双方都没讲透)

尽调揭出的根本选择:**"智能"从哪来?**

- **选项 A — BYO-agent(脚手架定位)**:贴近现状。产品是"运行时脚手架",用户带自己的 LLM/agent 消费 prompt。轻、快、本地优先,但卖点是"工具"。
- **选项 B — server 补脑(大脑定位)**:server 端调 LLM 做决策/语义 verify。这是 Codex 隐含方向,卖点强("更可靠可审计的 agent"),但要新建一整层、且与"本地优先/数据主权"的开源叙事有张力(运行事件要回传)。

这个选择决定产品是"脚手架"还是"大脑",也决定 Phase 3 的工作量量级。其余分歧已被事实消解,**只有这一个需要你定方向**。
